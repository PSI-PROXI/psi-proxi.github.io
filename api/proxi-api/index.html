Proteomics Expression Interface Working Group

A simple and efficient way of exchanging protein expression data using distributed repositories

PROXI API


The ProteomeXchange API will allow the interoperable exchange of protein expression information across multiple organizations and on multiple platforms. This is a freely available open standard for interoperability, that uses common web protocols to support serving and sharing of data on protein/peptide sequences, post-translational modifications and expression information. The API is implemented as a webservice to create a data source which may be integrated into visualization software, web-based proteomics portals or processed as part of proteomics analysis pipelines. It overcomes the barriers of incompatible infrastructure, databases and proteomics resources to enable protein expression data providers and consumers to better share proteomics data and work together on a global scale.




One API: Multiples Resources and Apps





This v0.0.1 of the PROXI API has been developed using JSON schema to model the data. The documentation presented here is provided as a freely available blueprint for software developers to take and implement alongside those already at the ProteomeXchamge partners.

The PROXI model comprises different views for each resource: - Protein - Dataset


Protein View

It can be considered as the result of the aggregation of different experiments, assays or MS-runs (in the case of MS-based proteomics), using the protein as the central point for the aggregation. For example, PeptideAtlas allows the users to retrieve the information for a specific protein identified in different experiments.

Protein view is the most widely used view for all the protein expression resources such as MOPED, GPMDB, CPTAC Assay Portal amount others. If proteins are not present (e.g. peptide oriented resource), the central view can be the peptides.


Dataset View

The dataset view is close the the model view in GPMDB. The dataset view is a project oriented view of each biological entity such as proteins, peptides, modifications. The dataset view enables to share data on specific projects opening the possibilities for each resources for re-analisis, data sharing and validation of the original results.

Dataset view is widely used in those resources such as MassIVE and PRIDE Archive that provides original datasets.




Common URL syntax




The first element of the URL is, of course, the location of the web service. The resource could have they own name, for best practice we suggest to finalize the name of the resource with proxi. Different resources can have their own web service and representation to access to their data and PROXI can be another view of these resources. The structure (resource/proxi/) allows an standardized url that will highlight to the end end-user that they are using the PROXI view and API of the resource.

VERSION

It is a common practice to have different specifications of the same API running in the same server. The PROXI implementation provide a versioning system that every resource can use for this purpose. With time, more versions might exist. The possible values for this bit are:

Version Description
v1.0 Version 1.0 of the PROXI API
v2.0 Version 2.0 of the PROXI API
current Will map always to the most current version



The version system correspond the PROXI API version and not to a database or builds versions. Some databases such as PeptideAtlas are based on builds which are like snapshots of the data related with the reference database used. In those cases we recommended to have different end-points for example:
1. www.peptideatlas.org/build1/proxi/v1.0/
2. www.peptidealtas.org/build2/proxi/v1.0/
